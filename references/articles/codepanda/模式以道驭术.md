程序猿的进阶基本功，模式。从一个熟练的程序猿进阶中级程序猿的分界线。

程序猿大部分工作中，都是拧螺丝。增删改查的处理业务逻辑代码，占据工作的90%，然后在没有任何设计模式的基础的知识积累，也能carry日常大部分的work。进而有些忽略了基本功的锻炼
--模式的学习。很多人的模式知识，应该是都是停留在面试时候，以及工作中的‘嘴炮’模式。

模式,应该像是武功秘籍的内功招式。设计模式就那么一些，在适合的业务场景下，选用合适的招数。久而久之，就做到了熟能生巧，炉火纯青，应用自如的自己的本能了。尤其是在各种市面的框架设计中，都是各种模式应用的体现。

设计模式，应该是出于对代码的业务层的抽象总结得出来的结果。对不同的业务场景下的各种不同的情况，选用不同的代码实现方式。其本质还是抽象，封装。原则，遵循对拓展开放，对改变关闭。通过封装跟抽象，继承等手法，尽可能达到代码的松耦合，高内聚。

入门的设计模式：简单工厂模式、工厂方法模式、抽象工程方法模式。理解这三个概念对设计模式的原理有着较好的解释。

只要是涉及到工厂两个字，其核心。就是客户端，不再直接实例化具体的对象。而是将实例化具体对象的操作，转交给工厂来实现。而不同的工厂模式，其实现的细节跟运用场景略有不同。

最简单的就是简单工厂模式，这个主要是解决客户端跟服务端的代码的分离。对客户屏蔽实例化对象的细节，再进行对象拓展修改的时候，不用对所有实现该实例化对象的客户端代码进行修改。

原始代码：

客户端：

$product = new Aproduct();

简单工厂模式：

客户端：

$product = Factory::create('A');

服务端：

public static class Factory(){

public static creaet($type){

swithc($type){

case A:

return new Aproduct();

break;

default:

return 'worng type!';

简单工厂模式，就是客户端不再实例化具体对象，而把这个new的操作都隐藏在了工厂中间。而这里可以通过工厂type类型，来实例化不同的具体类。让客户端不跟具体的对象打交道。从业务上实现所谓的隔离。

工厂方法模式，其实是一种便于拓展的模型。一种对于服务端更加松耦合的模型。简单工厂模式的话，在工厂类里面，集合了业务逻辑，工厂承担了对不同的type来实例化不同的具体类的使命。（当然这里可以通过命名规范，例如PRS4的命名空间的规范类似，将这里选择的实例化具体的对象进行编排。这样对对象名有较强的规范要求）。工厂方法模式，就是通过一个接口，对外提供一个必须的抽象的创建方法的接口。从而达到代码的可以拓展性以及封装。

工厂方法模式：

客户端：

$Afactory = new Afactory();

$product = $Afactory->produce();

服务端：

interface Factory(){

abstract produce();

}

class Afacotry implement Factory(){

public produce(){

return new Aproduct();

}

}

这里主要是针对单一产品进行实例化。通过不同的实例化工厂来达到对真正产品实例化隔离效果。

抽象工厂模式，虽说是一种进阶模型。但是个人感觉同样是存在对应的业务应用场景而生的。主要是针对产品组的生成。以上都是单一产品生成。而抽象工厂模式，是在工厂方法模式上的横向拓展，将相关联的产品实例化都收纳进来，从而达到代码封装的效果。

$factory = new Afactory();

$productA_1= Afactory->produceA();

$productA_2 = Afactory->produceB();
