#  谁动了我的数据？：并发的混乱

原创  功夫熊猫盼  功夫熊猫盼  [ 码农世界CodePanda ](javascript:void\(0\);)

_2024年01月26日 16:07_ __ _ _ _ _ _ 北京  _

收到一个线上报警邮件
，提示更新公共邮箱密码失败后，我像一名侦探一样开始了我的调查之旅。首先，我审视了那条可能是“罪魁祸首”的SQL更新语句。这条语句只有在更新失败时，返回结果为0，才会触发报警邮件。

就像这样：

*   *   *   *   *   *   *   *   *

if (Email::where('id', $this->email_id)->update(['status' => Email::STATUS_OPENED, 'password' => $pwd])) {
// 一切正常，继续执行
} else {
// 发生异常，拉响警报
}

我的第一个嫌疑人是那个不见踪影的邮箱ID  ，但调查日志显示，它不仅存在，而且在本地环境中表现得无可挑剔。

接下来
，我开始怀疑线上环境中是否存在一些版本的小幽灵，导致数据未变更时，返回的影响行数为0。但是，我的测试脚本证明了我的这个猜想是错误的。不管数据变没变，只要邮箱ID在，影响行数就是1。

这个问题似乎把我逼进了死胡同
。既不是SQL逻辑的问题，也不是数据的问题，那只能是代码逻辑的小精灵作怪了。通过进一步的调查，我发现这段代码是在Laravel框架的异步队列中执行的。这让我想到，也许是多个消费者在争夺同一块蛋糕（也就是任务），导致了冲突。

我尝试联系DBA来通过数据库日志支持我的猜想  ，但遗憾的是，DBA那边没有能够提供帮助的日志。

不过  ，当我观察服务器上的进程时，发现了同时存在的多个消费者任务进程。这就像在现场找到了多个相同的指纹，几乎可以确认犯罪嫌疑。

线上的消费者进程是通过Supervisor来保持常驻的
，而Supervisor的配置文件中，我发现了同时启动了多个进程的证据。在本地模拟这个配置后，问题稳定复现，这就像  最后  一块拼图被放到了正确的位置。

* 问题分析：

* 如果这条记录的ID确实是存在，那么无论status和password字段之前的值是什么，这条UPDATE语句都会被认为是成功执行了，并且影响了一行。即使status已经是opened，password也已经是*，执行这条语句数据库还是会返回1，因为它执行了更新操作。

* 但是，当多个进程或线程尝试同时更新同一行数据时，数据库的并发控制机制会介入，确保数据的一致性和完整性。通常情况下，只有第一个获得锁的进程可以更新数据，而其他的进程则要么等待锁的释放，要么在无法获取锁时操作失败。如果第一个进程成功更新了数据，它会返回1，因为数据被更改了。如果此时其他进程也尝试进行更新，它们可能会得到一个返回值0，这通常表示没有行被更新，因为第一个进程已经更改了数据，其他进程的更新操作并未引起任何数据变化。

* 这种差异通常是由数据库的隔离级别和锁机制决定的。在某些数据库系统和隔离级别下，如果一个进程尝试更新一行数据，而这行数据已经被另一个进程锁定，那么它可能会等待锁的释放，然后再执行更新，此时如果数据未发生变化，它仍然会返回1，因为它执行了更新操作。其他情况下，如果等待超时或不允许等待，它可能会立即返回0，表示没有更新任何数据。

在这种任务队列中
，如果没有合适的任务调度，启动多个进程就像是开了多个相同的派对，只会造成混乱。我通过调整Supervisor配置文件，把进程数改为1，就像是给派对定了一个入场券，问题得到了解决。

最后
，还有一个Supervisor的小窍门：在更改配置文件后，必须要执行reread命令来重新加载配置，否则就像是对着墙壁说话一样，没有任何效果。即使你使用supervisorctl
restart命令重启进程，没有reread，新的配置也不会被采纳。这个小细节可是大坑，千万不要掉进去。

这就是这次调查的结局  ，不仅解决了问题，还学到了一两招。就像是在看一部侦探剧，只  不过  这次的侦探是我自己。
